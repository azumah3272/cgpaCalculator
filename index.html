  • Without prototypes, functions can only call other functions that are already fully defined above them in the file — which would force an awkward, fragile ordering.

  • Only the types in a prototype have to match the real definition exactly — parameter names are just for readability and can even differ.
Concept 04

The "ask → check → re-ask" pattern

main.cpp
string inputStudentID(int uniChoice)
{
    string id;
    boolA plain flag that starts false and only flips to true once the input passes the check. This is the switch the loop watches. valid = false;

    whileThe loop body runs at least once, and keeps repeating for as long as valid stays false — meaning it will not let the program move on until it gets acceptable input. (!valid)
    {

    }
    return id;
}
  • This exact shape — a boolean flag, a while (!valid) loop, ask, check, re-ask — repeats across four different functions: selectUniversity, inputStudentID, selectCourses, and assignGrades.
  • It's the simplest reliable way in beginner C++ to guarantee "don't proceed until the input is actually usable," without exceptions or more advanced error handling.
  • The flag is deliberately named for what it means (valid), not how it's used — that's what makes while (!valid) read almost like English.
  • Notice the checking logic itself (validateStudentID) is a separate function — the loop's job is only to keep asking, not to know how to judge the answer.

String operations behind ID validation

                return false;
        }

  • .length(), .substr(start, count), and isdigit() are the three string tools doing all the work here.

Concept 06

cin >> vs getline

main.cpp
int selectUniversity()
{
    int choice;
    cin >> choice;   // leaves a newline behind in the buffer
    return choice;
}

string inputName()
{
    string name;
    cout << "Enter your name: ";
    cin.ignore();Clears that leftover newline character sitting in the input buffer from the previous cin >> choice. Without this line, getline would immediately read an empty leftover line instead of waiting for real input.
    getline(cin, name);Unlike cin >>, getline reads an entire line — including spaces — which is essential for a full name like "Kwame Asante".
    return name;
}
  • cin >> stops reading at the first whitespace — fine for a single word, a number, or an ID with no spaces, but it would chop "Kwame Asante" down to just "Kwame".
  • getline(cin, name) reads the whole line, spaces included — that's why the name field uses it instead.
  • The catch: cin >> leaves the newline character (from hitting Enter) still sitting in the input buffer. If getline runs right after, it reads that leftover empty newline instead of waiting for the user.
  • cin.ignore() discards exactly that leftover character, so the next getline starts clean.

Concept 08

How data flows between functions

  • selectUniversity() → returns uniChoice, which feeds into inputStudentID() AND getCourseList() AND later displaySummary().
  • getCourseList(uniChoice) → returns allCourses, which feeds into selectCourses().
  • selectCourses(allCourses) → returns selected, which feeds into assignGrades().
  • assignGrades(selected) → returns graded, which feeds into BOTH calculateCGPA() and displaySummary().
  • Nothing is a global variable. Every piece of information travels strictly through parameters going in and return values coming out.
  • This is why main() barely contains any logic of its own — it just names the sequence of steps, one line per function, each line's result becoming the next line's ingredient.
This is the payoff of designing function signatures carefully up front: main() reads almost like a sentence describing the program, not a tangle of logic.

Concept 10

Pass-by-reference in scoreToGrade

main.cpp
void scoreToGrade(double score, string &letterGradeThe & means this parameter is a REFERENCE — it isn't a copy. Changes made to letterGrade inside this function directly change the original variable the caller passed in., double &gradePointSame idea — a second reference parameter, letting this one function fill in TWO outputs at once, without needing two separate return values.)
{
    if (score >= 80 && score <= 100)
    {
        letterGrade = "A";
        gradePoint = 4.00;
    }
    // ...
}

// called like this:
string letterGrade;
double gradePoint;
scoreToGrade(score, letterGrade, gradePoint);
// letterGrade and gradePoint are now filled in — no return needed
  • Normally, a C++ function only gets copies of the values you pass it (pass-by-value) — changing a copy inside the function has zero effect on the caller's original variable.
  • The & in string &letterGrade changes that: it's a reference, meaning the function is working with the caller's actual variable, not a copy.
  • This is exactly why scoreToGrade can be void — instead of returning one value, it reaches back and updates two variables directly through its reference parameters.
  • Without references here, you'd need some other mechanism to return two values at once (like bundling them in a struct) — references are the simplest tool for this in beginner C++.

Concept 11

Why if/else-if here, but switch elsewhere

main.cpp
// continuous RANGE of scores → if/else-if chain
if (score >= 80 && score <= 100) { letterGrade = "A"; }
else if (score >= 75) { letterGrade = "A-"; }
else if (score >= 70) { letterGrade = "B+"; }
// ...

// discrete MENU number (an earlier design) → switch fit better
switchswitch compares one value against an exact list of fixed options — perfect for a 1-to-10 menu choice, but it can't express "any number from 75 to 79" in a single case. (gradeChoice)
{
    case 1: return 4.00;
    case 2: return 3.75;
}
  • switch matches a value against an exact list of discrete options — great for something like a 1–10 menu choice.
  • A raw score (0–100) needs ranges, not exact matches — "any number from 75 to 79 is an A-" can't be written as a single switch case.
  • if / else if chains handle ranges naturally because each condition is a full boolean expression (score >= 75), not just a fixed value to match.
  • Order matters in the chain: checking from highest to lowest means each else if only needs a lower-bound check — by the time you reach it, you already know the score failed every higher bracket above it.
Concept 12

Signaling errors with a sentinel value

main.cpp (earlier version)
double gradeToPoint(int gradeChoice)
{
    switch (gradeChoice)
    {
        case 1: return 4.00;
        // ... cases 2-10 ...
        default: return -1.0;-1.0 is impossible as a real grade point (they only range 0.00 to 4.00) — so this value can only mean "something went wrong," making it safe to check for.
    }
}

// caller checks for it:
double point = -1.0;
while (point < 0This condition is ONLY true if gradeToPoint returned the sentinel — any real grade point is 0.00 or higher, so this cleanly detects "invalid input" without any special exception-handling machinery.)
{
    // re-prompt...
}
  • A sentinel value is a special return value chosen specifically because it can never occur as a legitimate answer — here, -1.0, since real grade points only run from 0.00 to 4.00.
  • This gives the calling code a simple way to detect "that input was invalid" just by checking if (point < 0) — no crash, no exceptions, no extra machinery.
  • It's a common beginner-friendly pattern before a course introduces proper exception handling (try/catch) — simple, readable, and effective for small programs like this.
  • Risk to know about: sentinel values only work if you're certain about the valid range. If grade points could ever legitimately go negative, this trick would silently break.

Concept 14

The GPA formula, worked through

main.cpp
double calculateCGPA(vector<Course> selectedCourses)
{
    double totalQualityPoints = 0.0;
    int totalCreditHours = 0;

    for (int i = 0; i < selectedCourses.size(); i++)
    {
        totalQualityPoints += selectedCourses[i].creditHours * selectedCourses[i].gradePoint;Example: MATH103, 3 credit hours, grade B (3.25 points) → adds 3 × 3.25 = 9.75 to the running total.
        totalCreditHours += selectedCourses[i].creditHours;
    }

    ifGuards against dividing by zero — which would happen if, somehow, no courses were ever selected. Division by zero on doubles doesn't crash in C++, it produces "inf" or "nan", which would print as garbage in the final report. (totalCreditHours == 0)
        return 0.0;

    return totalQualityPoints / totalCreditHours;
}
  • The formula: GPA = Σ(Credit Hours × Grade Point) ÷ Σ(Credit Hours) — exactly matching the official calculation method.
  • Two accumulators build up across the loop: totalQualityPoints (the numerator) and totalCreditHours (the denominator).
Worked example — 3 courses:
MATH103 (3 CH, B = 3.25) → 9.75
CSSD111 (3 CH, A = 4.00) → 12.00
ENGL171 (2 CH, A- = 3.75) → 7.50
GPA = (9.75+12.00+7.50) ÷ (3+3+2) = 29.25 ÷ 8 = 3.66
Concept 15

Turning a CGPA into a class of degree

main.cpp
string classifyDegree(double cgpa)
{
    ifChecked from the highest boundary down — by the time we reach a lower else-if, the CGPA has already failed every higher check above it, so only the lower bound needs testing. (cgpa >= 3.60)
        return "First Class Honours";
    else if (cgpa >= 3.00)
        return "Second Class Honours (Upper Division)";
    else if (cgpa >= 2.50)
        return "Second Class Honours (Lower Division)";
    else if (cgpa >= 2.00)
        return "Third Class Honours";
    else if (cgpa >= 1.50)
        return "Pass";
    else
        return "Fail (No Award)";
}

  • A raw number like 3.42 doesn't mean much on its own — students and employers think in terms of class of degree, so this translates the GPA formula's output into that familiar language.
  • Same if/else-if shape as scoreToGrade, just with different boundaries and coarser categories (six bands instead of ten).
  • Bands used here follow the common Ghanaian GPA-based university scheme (UG, UCC-style): First Class 3.60–4.00, Second Upper 3.00–3.59, Second Lower 2.50–2.99, Third Class 2.00–2.49, Pass 1.50–1.99, Fail below 1.50.
  • Worth double-checking against your own course handbook — some institutions use slightly different cutoffs (e.g. a 2.00 floor for Second Lower instead of 2.50), so exact boundaries can vary by school and by year.

Concept 15

The ternary operator

main.cpp
void displaySummary(...)
{
    string universityName = (uniChoice == 1) ? "University of Ghana (Legon)" : "GCTU";Read as: "if uniChoice equals 1, use the first string; otherwise use the second." One line instead of a four-line if/else block.
}
  • The ternary operator condition ? valueIfTrue : valueIfFalse is shorthand for a simple if/else that just picks between two values.
  • It reads left to right: check the condition, then pick one of the two results — assigned directly into universityName in a single statement.
  • Limitation: it only works cleanly for exactly two outcomes. If a third university were added, this line would need to become a real if / else if / else chain instead — ternaries don't scale well past two branches.
Concept 16

fixed and setprecision(2)

main.cpp
cout << "Final CGPA: " << fixed << setprecision(2)fixed forces decimal notation instead of switching to scientific notation for certain values. setprecision(2) then locks the output to exactly 2 decimal places. << cgpa << "\n";

// without these manipulators, cgpa might print as:
// 3.41999999999999    or    3.4    or    3.42e+00
// with them, it always prints as:
// 3.42
  • Raw double values don't print consistently — floating-point rounding can produce ugly trailing digits, and very large or small numbers can switch to scientific notation automatically.
  • fixed (from <iomanip>) locks output to plain decimal notation.
  • setprecision(2) then caps it to exactly 2 digits after the decimal point.
  • Sticky behavior: once applied to cout, both settings persist for every subsequent output in the program — which is exactly what's wanted here, since every grade point and the final CGPA should look consistent.
Concept 17

The signed vs. unsigned comparison warning

main.cpp
if (courseChoice < 1 || courseChoice > allCourses.size()courseChoice is an int (signed). allCourses.size() returns size_t (unsigned). Comparing signed against unsigned can trigger a compiler warning, even though the actual values here are always non-negative in practice.)

// a stricter compiler might flag this with -Wsign-compare
// silencing it, if desired:
courseChoice > static_cast<int>(allCourses.size())
  • .size() on a vector returns size_t, an unsigned type — it can never represent a negative number.
  • courseChoice is a plain int — a signed type, which can represent negatives.
  • Comparing a signed value against an unsigned one can trigger a compiler warning (-Wsign-compare) on stricter settings, because the compiler can't always guarantee the comparison behaves as expected across all possible values.
  • In this program it works correctly in practice, since course counts are never negative — but the clean fix, if a lecturer asks, is wrapping the unsigned side in static_cast<int>().
Concept 18

Format validation, not real authentication

  • The ID check only confirms a string looks like a real ID of the invented format for that university — length, prefix, and digit pattern.
  • It does not verify the person is actually enrolled anywhere, and it can't check against a real student database (this program has none).
  • Any string matching the pattern — real or made up — will pass. That's an intentional, honest scope limit for a console-based academic exercise, not an oversight.
If asked directly: "This validates format, not identity. A real authentication system would need a database lookup and typically a password or PIN, which is out of scope for this program."
Concept 19

What was left out, and what could be added

  • Left out deliberately: non-numeric grade categories (X = Absent, Z = Disqualified, IC = Incomplete, AD = Audit) — these don't fit "enter a score you achieved," since they represent situations where no numeric assessment happened.
  • Left out deliberately: real persistence — nothing is saved to a file or database; every run starts fresh.
  • Left out deliberately: real authentication — the ID check is format-only, as covered on the previous slide.
  • Possible extensions: save each run's summary to a text or CSV file; support more universities by generalizing the course-list and ID-format logic; track a running CGPA across multiple semesters instead of one semester's GPA at a time.
